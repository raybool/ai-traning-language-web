# AGENTS.md

## Общие правила проекта

Проект разрабатывается на Next.js, React, TypeScript и SCSS Modules.

При генерации, изменении или рефакторинге кода необходимо соблюдать текущую архитектуру проекта, стиль импортов, нейминг файлов и существующие паттерны.

Не создавать лишние файлы, абстракции и переиспользуемые сущности без необходимости.

---

# Структура компонента

Каждый компонент должен располагаться в отдельной папке.

Базовая структура папки компонента:

```txt
ComponentName/
  index.ts
  ComponentName.tsx
  ComponentName.module.scss
  types.ts
  constants.ts
  utils/
  components/
```

Не создавать пустые файлы и папки, если они не используются.

---

# Назначение файлов внутри компонента

## index.ts

Файл экспортирует только те сущности, которые должны использоваться за пределами папки компонента.

Пример:

```ts
export { ComponentName } from './ComponentName';
```

Не экспортировать внутренние `utils`, `constants`, `types` и локальные компоненты, если они не являются частью публичного API компонента.

---

## ComponentName.tsx

Основной файл компонента.

Правила:

- компонент должен быть небольшим и читаемым;
- бизнес-логику выносить в `utils`;
- локальные дочерние компоненты выносить в `components`;
- типы props хранить в `types.ts`;
- стили подключать из `ComponentName.module.scss`.

Пример:

```tsx
import { FC } from 'react';

import styles from './ComponentName.module.scss';
import type { TComponentNameProps } from './types';

export const ComponentName: FC<TComponentNameProps> = ({ title }) => {
  return <div className={styles.root}>{title}</div>;
};
```

---

## ComponentName.module.scss

Файл стилей компонента.

Правила:

- использовать SCSS Modules;
- главный класс компонента называть `root`, `page`, `wrapper` или по смыслу;
- не использовать глобальные стили без необходимости;
- не писать стили, которые должны быть частью общего UI-kit.

Пример:

```scss
.root {
  display: flex;
  flex-direction: column;
  gap: 16px;
}
```

---

## types.ts

Файл с типами, относящимися только к текущему компоненту.

Правила:

- props компонента называть через `TComponentNameProps`;
- не хранить здесь глобальные типы;
- если тип используется во многих местах проекта, вынести его в общий слой.

Пример:

```ts
export type TComponentNameProps = {
  title: string;
};
```

---

## constants.ts

Файл с константами, относящимися только к текущему компоненту.

Правила:

- хранить только локальные константы;
- не выносить сюда данные, которые должны приходить из API, store или props;
- глобальные константы должны находиться в общем слое проекта.

Пример:

```ts
export const DEFAULT_TITLE = 'Default title';
```

---

## utils/

Папка с дополнительными функциями, которые нужны только текущему компоненту.

Правила:

- хранить только чистые функции;
- не размещать React-компоненты;
- если функция нужна в нескольких местах проекта, вынести её в общий слой.

Пример:

```txt
utils/
  formatDate.ts
  getVisibleItems.ts
```

---

## components/

Папка с локальными дочерними компонентами.

Использовать только для компонентов, которые:

- нужны только текущему компоненту или странице;
- не переиспользуются за пределами текущей папки;
- помогают разгрузить основной файл компонента.

Пример:

```txt
components/
  EmptyState/
    index.ts
    EmptyState.tsx
    EmptyState.module.scss
    types.ts
```

Если компонент можно переиспользовать в других местах проекта, не хранить его в локальной папке `components/`.

# Правила создания новых компонентов

При создании нового компонента необходимо:

1. Создать отдельную папку с названием компонента.
2. Создать основной файл компонента `ComponentName.tsx`.
3. Создать `index.ts` для публичного экспорта.
4. Создать `ComponentName.module.scss`, если компоненту нужны стили.
5. Создать `types.ts`, если у компонента есть props или локальные типы.
6. Создать `constants.ts`, если есть локальные константы.
7. Создать `utils/`, если есть локальные вспомогательные функции.
8. Создать `components/`, если есть локальные дочерние компоненты.

Не создавать пустые файлы только ради структуры.

---

# Правила импортов

Импорты должны быть сгруппированы в следующем порядке:

1. React и сторонние библиотеки.
2. Импорты из алиасов проекта.
3. Относительные импорты.
4. Стили.
5. Типы, если они не были импортированы выше.

Пример:

```tsx
import { PrimaryButton } from '@/components/ui/PrimaryButton';
import { FC } from 'react';

import styles from './ComponentName.module.scss';
import type { TComponentNameProps } from './types';
```

Если в проекте уже используется другой порядок импортов, следовать существующему стилю.

---

# TypeScript

Правила:

- использовать строгую типизацию;
- не использовать `any` без необходимости;
- для props использовать отдельный тип;
- типы props называть `TComponentNameProps`;
- интерфейсы использовать только там, где это уже принято в проекте;
- не дублировать типы, если они уже существуют.

Пример:

```ts
export type TUserCardProps = {
  name: string;
  age?: number;
};
```

---

# React

Правила:

- использовать функциональные компоненты;
- компонент объявлять через `FC`, если это соответствует стилю проекта;
- избегать лишнего `useMemo`, `useCallback`, `memo`;
- использовать мемоизацию только при реальной необходимости;
- не выносить простую JSX-разметку в отдельный компонент без причины;
- не хранить вычисляемые значения в `useState`, если их можно получить из props или store.

---

# Next.js

Правила:

- использовать `'use client'` только там, где действительно нужны клиентские хуки, обработчики событий, browser API или состояние;
- если компонент использует `useSearchParams`, оборачивать его в `Suspense` на уровне страницы;
- не превращать server component в client component без необходимости;
- не использовать `window`, `document`, `localStorage`, `sessionStorage` без проверки клиентского окружения.

---

# Redux и селекторы

Правила:

- простые селекторы писать обычными функциями;
- `createSelector` использовать только для производных данных или тяжёлых вычислений;
- не возвращать новый объект или массив из селектора без необходимости.

Плохо:

```ts
export const selectEntity = createSelector([selectEntityState], (entity) => ({
  ...entity,
}));
```

Хорошо:

```ts
export const selectEntity = (state: RootState) => state.entity;
```

---

# SCSS Modules

Правила:

- использовать локальные классы из `*.module.scss`;
- не использовать глобальные селекторы без необходимости;
- использовать понятные имена классов;
- избегать чрезмерной вложенности.

Пример:

```scss
.card {
  padding: 24px;
  border-radius: 16px;
}

.title {
  margin: 0;
}
```

# Экспорт компонента

Каждый компонент должен экспортироваться через `index.ts`.

Пример:

```ts
export { BlockedPageContent } from './BlockedPageContent';
```

Импортировать компонент нужно через папку:

```ts
import { BlockedPageContent } from '@/components/pages/BlockedPageContent';
```

Не импортировать напрямую из файла:

```ts
import { BlockedPageContent } from '@/components/pages/BlockedPageContent/BlockedPageContent';
```

---

# Utils

Функции из `utils/` должны быть:

- чистыми;
- легко тестируемыми;
- без зависимости от React;
- без работы с DOM;
- без прямой работы со store.

Пример:

```ts
export const formatBlockedUntil = (value: string | null): string | null => {
  if (!value) return null;

  const date = new Date(value);

  if (Number.isNaN(date.getTime())) return null;

  return date.toLocaleString();
};
```

---

# Что нельзя делать

Не нужно:

- создавать пустые файлы;
- создавать универсальные компоненты раньше времени;
- экспортировать внутренние детали компонента;
- использовать `any`;
- добавлять `useMemo` и `useCallback` без причины;
- делать component client component без необходимости;
- смешивать бизнес-логику, форматирование данных и JSX в одном большом файле.

---

# Перед завершением задачи

После изменения кода необходимо проверить:

- компонент соответствует структуре проекта;
- `index.ts` экспортирует только публичный API;
- нет пустых файлов;
- нет лишнего `createSelector`;
- нет лишнего `'use client'`;
- нет лишней мемоизации;
- импорты соответствуют стилю проекта;
- TypeScript-типы корректны;
- стили находятся в SCSS Module.
